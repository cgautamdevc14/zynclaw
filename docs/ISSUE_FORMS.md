# Issue Forms

Issue forms make intake easier for teammates who do not want to edit files.

## Available Forms

| Form | Use For |
|------|---------|
| Product intake | New ideas, requirements, and first slices |
| Local agent task | Scoped work that may be assigned to an agent |
| Marketing request | Positioning, campaign, content, or launch work |
| Sales enablement | ICP, talk track, demo, outreach, or objections |
| QA report | Validation results and release recommendation |
| Connection request | New tool connections and permission reviews |

## What To Include

Good issues include:

- Objective.
- Audience or user.
- Scope.
- Non-goals.
- Acceptance criteria.
- Validation or review owner.

## Why The Forms Are Structured

Structured issue forms reduce missing context. GitHub supports required fields,
labels, dropdowns, text areas, and checkboxes, which makes intake more
consistent than blank issues.

## When To Use Files Instead

Use a work-item folder when the task spans multiple functions or needs a launch
handoff:

```bash
make scaffold WORK="short task name"
```
